## Database Schema

### Design

```sql
projects   (project_id PK)
subjects   (subject_id PK, project_id FK, condition, age, sex, treatment, response)
samples    (sample_id PK, subject_id FK, sample_type, time_from_treatment_start)
cell_counts(sample_id PK/FK, b_cell, cd8_t_cell, cd4_t_cell, nk_cell, monocyte)
```

### Rationale

**Normalization:** Patient-level attributes (condition, sex, treatment, response) are stored once in `subjects` rather than repeated on every sample row. This avoids update anomalies and keeps data consistent.

**Separation of concerns:** `cell_counts` is split from `samples` because cell populations are a distinct measurement domain. If new assay types are added (e.g. RNA-seq counts, protein levels), they can sit in separate tables without widening the samples table.

**Scalability considerations:**
- *Hundreds of projects / thousands of samples:* The normalized schema avoids redundant storage. Adding a `projects` metadata table (sponsor, indication, start_date) is trivial.
- *Diverse analytics:* Indexes on `condition`, `treatment`, `response`, `sample_type`, and `time_from_treatment_start` make common filter patterns fast. A read replica or columnar extension (e.g. DuckDB) could be layered on for heavier aggregations.
- *New cell populations:* If the panel grows significantly (e.g. 50+ markers from mass cytometry), a long-format `cell_counts` table (sample_id, population, count) would scale better than adding columns. The current wide format is optimal for the 5-population case.

## Code Structure

| File | Responsibility |
|------|---------------|
| `load_data.py` | Schema creation + CSV ingestion. Idempotent (drops and recreates the DB). |
| `analysis.py`  | Pure analytical functions (frequency table, Mann-Whitney tests, subset queries) + a `run_all()` entrypoint that saves all outputs to `outputs/`. |
| `dashboard.py` | Streamlit app that imports from `analysis.py` and renders interactive Plotly charts. Uses `@st.cache_data` so repeated navigation doesn't recompute. |

The analysis functions are deliberately separated from the dashboard so they can be unit-tested, run headlessly in CI, or called from other tools without starting a web server.

## Part 3 — Statistical Findings

Mann-Whitney U tests (two-sided, α = 0.05) were used because:
- Cell population percentages are not guaranteed to be normally distributed.
- Sample sizes per group can be small, making non-parametric tests more robust.

See `outputs/statistical_results.csv` for full results.
